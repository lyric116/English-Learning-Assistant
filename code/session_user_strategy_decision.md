# 会话与用户策略决策（P3-01）

Date: 2026-02-25  
Owner: Codex

## 1. 目标与范围
- 明确 MVP 到持久化架构阶段的身份策略边界。
- 定义匿名会话与登录用户在数据、权限、可见性上的职责分离。
- 为后续 `P3-02 ~ P3-09` 提供可执行前提，不在本步骤引入真实登录实现。

## 2. 当前基线（已实现）
- 前端匿名会话 ID：`client/src/lib/session.ts`
  - 首次访问生成 `anonymous-session-id`（`crypto.randomUUID` 优先，失败时 fallback）。
  - 存储于 localStorage，当前浏览器可恢复。
- 请求透传：`client/src/lib/api.ts`
  - 所有请求自动携带请求头 `x-anonymous-session-id`。
- 学习数据存储：五大模块核心历史目前仍以 localStorage 为主。

## 3. 策略决策

### 3.1 双轨身份模型
- `Anonymous Session`（匿名会话）
  - 标识：`anonymous-session-id`
  - 用途：无登录状态下维持单设备学习连续性
  - 数据可见性：仅当前浏览器（本地）
- `Authenticated User`（登录用户）
  - 标识：`user_id`（后续由登录系统签发）
  - 用途：跨设备数据一致性、长期持久化、账号级安全策略
  - 数据可见性：同账号授权设备

### 3.2 边界原则
- 匿名会话不承诺跨设备一致性。
- 登录用户数据以服务端为准，客户端缓存仅做加速层。
- AI 配置（API Key）默认不落服务端，除非后续引入受控加密托管能力。

## 4. 会话生命周期

### 4.1 匿名会话生命周期
1. 创建：首次访问生成 `anonymous-session-id` 并持久化。  
2. 使用：每次请求携带该 ID，用于服务端关联临时数据（后续 P3 阶段实现）。  
3. 刷新：浏览器刷新不变；清理本地存储后重新生成。  
4. 失效：本地清除、隐私模式结束、显式“重置学习数据”会触发会话重建。  
5. 恢复：若 key 存在且格式合法则直接复用。

### 4.2 登录会话生命周期（规划）
1. 创建：用户完成登录，获得访问令牌与刷新令牌（P3-02 细化）。  
2. 使用：请求以用户令牌鉴权，匿名 ID 作为可选迁移上下文。  
3. 刷新：短 token 过期后由 refresh 流程续期。  
4. 失效：主动登出、refresh 失效、服务端撤销。  
5. 恢复：仅在 refresh 有效窗口内可无感恢复。

## 5. 权限边界
- 匿名用户
  - 允许：五大模块学习功能、生成报告、客户端持久化历史。
  - 不允许：跨设备同步、账号级共享页面、长期服务端数据托管。
- 登录用户
  - 允许：跨设备同步、账号历史回放、后续协作/分享扩展能力。
  - 不允许：读取他人数据；越权访问未授权会话。
- 服务端安全边界
  - 必须校验会话/用户标识格式与来源。
  - 对匿名流量执行更严格的配额与速率限制。
  - 错误日志不得输出 API Key/token 原文。

## 6. 迁移策略（匿名 -> 登录）

### 阶段 A：标识并存（P3-02/P3-03）
- 新增用户主标识字段，同时保留匿名会话上下文。
- 数据模型支持 `owner_type`（anonymous/user）与 `owner_id`。

### 阶段 B：双写与回填（P3-06/P3-08）
- 登录后首个会话启动匿名数据回填任务：
  - flashcards / sentence / reading / quiz / report 按模块迁移。
- 过渡期执行“本地 + 服务端”双写，记录迁移状态位。

### 阶段 C：切换与收敛（P3-09 之后）
- 读取优先级切到服务端；本地仅保留最近缓存。
- 当回填完成且稳定后，逐步关闭匿名主路径的数据写入。

### 冲突处理规则
- 同一记录冲突默认“最近更新时间优先”。
- 无法自动合并的富文本字段（如笔记）保留双版本并提示用户手工选择。

## 7. 验证清单（P3-01 验收）
- [x] 会话生命周期已覆盖创建、使用、刷新、失效、恢复。
- [x] 权限边界已区分匿名与登录用户能力。
- [x] 迁移策略已定义阶段、双写与冲突规则。
- [x] 输出可直接作为 P3-02 会话令牌流转设计输入。

## 8. 对后续步骤的约束
- P3-02 的令牌流转不得破坏匿名体验（未登录用户必须可继续使用）。
- P3-06/P3-07 迁移顺序应遵循“先低风险模块后核心链路模块”。
- P4+ 的日志与监控需同时追踪 `anonymous_session_id` 与 `user_id`（如存在）。
