# English Learning 改造实施计划（implement_plan）

适用范围：基于 `agent.md` 的模块拆解与改造路线，将项目从“演示型”推进到“可持续学习产品”。  
执行原则：每次只完成一个步骤；每个步骤结束必须执行该步骤的验证测试；测试未通过不得进入下一步。  
限制要求：本文档只给执行指令，不包含实现代码。

## MVP 裁剪规则（lyricx 确认）

1. 不必完整执行 65 步，可按 MVP 目标裁剪。
2. Phase 可重排/并行，但依赖顺序固定：
   - 先做 `Settings / AI 网关 / 基础路由`
   - 再做单模块闭环（页面 + API + 存储）
   - 最后做部署与质量门槛
3. “评审通过/负责人确认”统一由 lyricx 确认。
4. 交付格式：每步一个 git commit（或 PR）；附 checklist（功能截图、复现步骤、跑过命令）。
5. Gate C 统一解释为：
   - `npm run lint`（前后端）
   - `npm test`（至少后端；前端有测试则运行）
   - `npm run build`（前后端或根目录）
   - smoke：启动 dev 后至少一次 curl/页面访问
6. Step 57-59 测试栈固定：
   - 前端单测：Vitest + React Testing Library
   - 后端测试：Vitest + Supertest
   - E2E：Playwright（仅 1-2 条关键路径）
7. AI 相关自动化测试默认必须 mock；可提供真实模型手动 smoke 脚本，但不进 CI、不作为 Gate 必过项。
8. MVP 阶段不引入数据库，默认 localStorage/IndexedDB；如必须引入持久化，SQLite 优先。
9. MVP 用户体系仅匿名会话（本地存储），登录与第三方登录延后到 Phase 3+。
10. 部署目标保持 `Vercel + Serverless`，先保证可部署可运行。
11. 性能/稳定性（MVP）：
   - 非 AI 接口 P95 < 300ms（本地简单接口）
   - AI 接口不设硬阈值，但必须有 loading/超时/错误提示
   - 功能路径可用，错误可读且可恢复
12. 安全边界（MVP）：
   - 不在服务端落库用户 API key
   - 日志脱敏，不打印 key，不打印完整 prompt/完整模型返回
   - 数据默认本地保存
   - `baseUrl` 做协议/内网限制基础校验
13. 可访问性对齐 WCAG 2.1 AA 基础项：键盘可操作、focus 可见、表单有 label、基础对比可读。
14. UAT（Step 64）通过标准：
   - 关键用例通过率 >= 95%
   - 10 分钟内可回滚至上一稳定版本（Vercel 回滚或回退 commit）
15. 成本约束（MVP）：
   - 默认每用户每天 <= 50 次 AI 调用（前端计数与提示）
   - 超额后允许手动继续或限制（实现时二选一）
16. `architecture.md` / `progress.md` 本轮不做初始化。

## Phase 0：基线与目标品牌（Step 1-8）

1. [P0-01] 建立实施分支与任务追踪文档。  
实施指令：从当前主线创建独立改造分支；在 `code/` 下创建实施追踪文档，记录每一步状态、执行人、时间。  
验证测试：`git branch --show-current` 返回新分支名；追踪文档包含 Step 1 条目。  
完成标准：分支和追踪文档均可用。

2. [P0-02] 安装并确认前后端依赖可用。  
实施指令：分别在根目录、`client/`、`server/` 确认依赖完整且无安装错误。  
验证测试：依赖安装命令无报错；`npm ls --depth=0` 在根目录、前后端均可执行完成。  
完成标准：开发环境可稳定启动。

3. [P0-03] 建立当前质量基线。  
实施指令：执行当前仓库约定的最小质量门禁命令并记录结果。  
验证测试：`npm run build` 成功；`cd client && npm run lint` 成功。  
完成标准：得到可复现的“改造前”质量基线记录。

4. [P0-04] 建立当前功能基线。  
实施指令：手工跑通五大模块主流程（闪卡、句子分析、阅读、测验、成就），记录关键路径与阻塞点。  
验证测试：每个页面都能进入并完成一次核心操作；记录中包含至少一个成功结果截图或日志描述。  
完成标准：形成“当前行为对照表”。

5. [P0-05] 建立后端可用性基线。  
实施指令：验证健康检查与 AI 配置测试接口可达，并记录失败场景的返回行为。  
验证测试：`/api/v1/health` 返回成功；`/api/v1/ai/test` 在有效配置和无效配置下都返回可解释结果。  
完成标准：后端基线行为明确。

6. [P0-06] 明确目标用户与核心使用场景。  
实施指令：在 `code/` 产出目标用户文档，至少覆盖 3 类用户（例如考试、职场、日常提升）及其学习目标。  
验证测试：文档含用户画像、使用频率、成功指标三部分；团队评审通过。  
完成标准：改造方向可被量化验证。

7. [P0-07] 定义品牌规范与文案语气。  
实施指令：确定产品名、主副色、排版语气、按钮文案风格、错误提示语气，并沉淀为品牌规范文档。  
验证测试：品牌文档包含色板、字体层级、语气示例；产品负责人确认。  
完成标准：UI 和文案改造有统一标准。

8. [P0-08] 对齐仓库文档与真实技术栈。  
实施指令：修正过期文档（重点是 `CLAUDE.md` 的旧架构描述），确保与 React + Express 现状一致。  
验证测试：`rg "Vanilla JavaScript|零依赖|无构建工具" CLAUDE.md` 不再匹配旧描述；README 与运行方式一致。  
完成标准：开发者文档不再误导。

11. [P1-03] 执行颜色对比度审查。  
实施指令：逐页检查主要文本与背景、按钮与背景的可读性，修正低对比组合。  
验证测试：关键文本与按钮达到可访问性对比要求；暗/亮主题都可读。

验证测试：所有页面的按钮 hover、active、disabled 状态一致；无单页私有样式残留。  
完成标准：按钮行为标准化。

13. [P1-05] 统一输入组件风格。  
实施指令：统一输入框、文本域、下拉框的边框、焦点态、错误态、提示态。  
验证测试：设置弹窗、闪卡输入、阅读输入的表单样式一致；错误态可见且文案明确。  
完成标准：表单体验稳定一致。

14. [P1-06] 统一反馈组件风格。  
实施指令：统一加载、空状态、错误提示、Toast、骨架屏的视觉与文案模板。  
验证测试：每个模块至少触发一次加载态和一次错误态；反馈组件无样式冲突。  
完成标准：用户能快速识别当前状态。

15. [P1-07] 重构导航与页面骨架。  
实施指令：统一 `Navbar`、`Footer`、`Layout` 的间距、层级与移动端折叠策略。  
验证测试：桌面端和移动端导航均可访问全部模块；设置入口可在两端稳定打开。  
完成标准：全站导航路径一致。

16. [P1-08] 改造首页视觉与信息结构。  
实施指令：首页强化产品定位、五大模块入口、学习路径引导与 CTA 层级。  
验证测试：首页首屏可在 5 秒内识别产品定位和下一步操作；移动端首屏无折叠错位。  
完成标准：首页可承担转化入口角色。

17. [P1-09] 统一五大模块页面框架。  
实施指令：统一每页的“输入区-结果区-历史区-操作区”结构，减少学习成本。  
验证测试：五大页面都具备一致的信息分区与操作顺序；新用户可在 1 分钟内完成首次操作。  
完成标准：跨模块体验一致。

18. [P1-10] 增加关键动效与响应式细节。  
实施指令：为页面进入、卡片翻转、结果出现添加有限且有意义的动效；补齐移动端点击热区。  
验证测试：动画无卡顿；移动端按钮可点击区域满足易用性要求；无布局抖动。  
完成标准：交互反馈更清晰但不过度。
